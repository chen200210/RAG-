2) 最体现工程深度的 3 个技术细节
Hybrid Search + RRF 风格融合排序（强抗“数值/关键词丢失”）

纯向量检索对“500万/48小时/六点/小时”等硬事实不稳定；BM25对关键词、数字更敏感。
你现在用“向量 topK + BM25 topK → 加权 RRF 合并去重排序”，把两路信号都用起来，显著提升数值题召回率与 Ragas 的 context_precision。
MD5 参数盐值（避免“改了 chunking 但索引没更新”的隐性灾难）

只用文本 MD5 会导致：你把 chunk_size/overlap 从 1000/100 改为 800/150，但系统以为“内容没变”直接跳过重入库。
现在在 knowledge.py 里把 chunk 参数拼进 MD5 计算（salt），保证切片策略一变就会触发重向量化，评估才“真生效”。
可观测性设计：Rerank 得分显性化 + 引用可追溯

final_docs 强制保留 relevance_score，前端 expander 直接展示得分，面试时你可以解释“我如何验证检索链路有没有变好”。
source/chunk_id 全链路传递，回答里的 [来源: 文件名-片段ID] 能在 UI 悬浮回看原文，属于“可解释 RAG”。
3) 《折叠北京》不使用 Rerank 时，最容易在哪个环节出错？为什么？
最容易出错的环节：粗排检索阶段（尤其是 topK 截断 + 排序不稳）。

原因拆解（面试里可以这样讲）：

长文档 + 数值/时间点信息往往是“局部硬事实”，语义上不一定和问题句子高度相似：
向量召回可能更偏“主题相关”（比如城市/折叠/阶层），但漏掉包含 500万/2500万/五千万/48小时/6点/22点这些关键数字的片段。

即使混合检索能召回到一些相关片段，没有 Rerank 时通常仍要在“前 N 条”里做截断：
一旦排序把真正含答案的片段排在 N+1，LLM 看到的 context 就缺失关键证据，按照你的严谨 prompt 就只能输出“知识库中未记载相关内容”。

跨段落因果/对话（老葛解释、老陆观点）对 chunk 边界很敏感：
如果粗排阶段拿到的是“被切断的一半对话”，LLM 很难完成“因果推断/观点对比”，也更容易判定“证据不足”。

一句话总结：不使用 Rerank 时，系统最容易在“粗排排序+截断”处把正确片段挡在 TopK 之外；而长文档数值题恰好是向量排序最不稳定的一类。
